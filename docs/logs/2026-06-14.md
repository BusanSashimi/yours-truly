## i18n foundation (next-intl) — Phase 0 + 1

**Goal:** Add EN/KO/JA internationalization, default English. After web research
the chosen stack is **next-intl v4.13** (de-facto standard for Next 15 App Router;
RSC-native, ~2 KB, built-in routing/middleware/hreflang). Routing: `localePrefix:
'as-needed'` (EN unprefixed, `/ko`,`/ja` prefixed) with `localeDetection` on.

User decisions (locked): translate **UI chrome only** (couples' invitation content
stays as-authored); invitation labels follow the **guest's browser locale** (uniform
`[locale]` routing, no per-invitation locale model); first-visit language **auto-detected
from Accept-Language**, remembered via cookie.

Phase 0 (setup):
- `pnpm --filter @yours-truly/web add next-intl`.
- `src/i18n/routing.ts` (defineRouting: locales en/ko/ja, default en, as-needed),
  `request.ts` (getRequestConfig, validates locale via `routing.locales.includes`,
  loads `../../messages/<locale>.json`), `navigation.ts` (createNavigation wrappers).
- Wrapped `next.config.ts` with `createNextIntlPlugin('./src/i18n/request.ts')`.
- `src/middleware.ts` (createMiddleware; matcher excludes `api`,`_next`,dotted files —
  critical so the `/api` rewrite + better-auth are untouched).
- `messages/{en,ko,ja}.json` placeholder catalogs (`Common.appName`).

Phase 1 (restructure):
- `git mv` both route groups under `app/[locale]/`; `reset.scss` + `favicon.ico` stay
  at `app/` root. The app has **two separate root layouts** (one per route group, each
  renders `<html>`) — preserved that; did NOT add an `app/[locale]/layout.tsx` (would
  nest `<html>`). Each root layout now: async, reads `params.locale`, validates →
  `notFound()`, `setRequestLocale(locale)`, exports `generateStaticParams`, `<html
  lang={locale}>`, wraps children in `<NextIntlClientProvider>`. Fixed reset.scss
  relative import `../` → `../../`.
- Swapped navigation in 8 files: `next/link` Link → `@/i18n/navigation` Link (all),
  `next/navigation` useRouter → `@/i18n/navigation` useRouter (5 client files).
  `notFound` deliberately left on `next/navigation` (locale-agnostic).

Verification (dev server, port 3000): `/`,`/samples`,`/login`→200 `lang="en"` unprefixed;
`/ko*`→`lang="ko"`, `/ja*`→`lang="ja"`; `/fr`→404; Accept-Language ko/ja on `/`→307 to
`/ko`/`/ja`. Confirmed `<Link>` emits locale-prefixed hrefs (`/ko/samples/…` on the ko
page, unprefixed on en). `tsc --noEmit` clean on source.

Gotcha fixed: the initial `pnpm add` ran from the wrong (outer) dir and created a stray
standalone `apps/web/pnpm-lock.yaml` (next-intl recorded only there, not the root
lockfile) → Turbopack "multiple lockfiles" warning. Deleted it and re-ran `pnpm install`

**Not yet done (next phases):** Phase 2 = extract hardcoded strings to `t()` keys +
