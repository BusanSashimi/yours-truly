## Prompt 1 — "check why I cannot access my EC2 instance via ssh after restarting it"

Investigated from the Mac via AWS CLI + direct probes.

- `aws sts get-caller-identity` — confirmed local creds = `user/busan-sashimi`.
- `aws ec2 describe-instances i-011a7e6377f9f0d18` — **running**, launched
  `2026-06-10T02:48:31Z` (the user had *restarted* it after the prior session
  stopped it), EIP `44.197.95.242` still attached, SG `launch-wizard-2`
  (`sg-0b7fa275a453b96d5`) unchanged.
- `aws ec2 describe-addresses` — EIP still associated → IP/DNS not the problem.
- SG ingress — port 22 open to `0.0.0.0/0` → firewall not the problem.
- `describe-instance-status` — system/instance checks **ok**, reachability **passed**.
- `nc -vz 44.197.95.242 22` — TCP **succeeds**.
- SSH banner grab + `ssh -vv` — **no banner; handshake hangs** (had to kill it).
- `curl https://44.197.95.242` — **times out** (site not served either).

**Conclusion:** Not an SSH/IP/SG/key problem. The instance is healthy at the AWS
layer but unresponsive at the application layer — the signature of severe
CPU/RAM exhaustion. On this host that's the cryptominer relaunching on the
restart (reboot persistence), re-reserving ~2.3 GB huge pages and pegging CPU, so
`sshd` accepts the TCP connection but can't fork a session. **Restarting the box
re-armed the malware.**
